## Engineering Notes

I like projects that are small enough to understand deeply and useful enough to keep improving. My default approach is to start with a working core, add tests around the important behavior, and write the README as part of the product rather than an afterthought.

When I publish code, I try to make the repository answer three questions quickly:

1. What problem does this solve?
2. How do I run it locally?
3. Where should I look first if I want to change it?

## Repo Standards

Most public repos here follow the same basic structure:

- A short README with the problem, quick start, and design notes.
- A minimal test suite covering the core behavior.
- CI configuration for the main branch.
- Small commits with focused messages so the project history is easy to scan.
